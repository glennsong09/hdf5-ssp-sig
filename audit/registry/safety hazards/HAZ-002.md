# HAZ-002: Concurrent access without safe coordination

**HDF5 SSP vulnerability tags:** `OPS|LIB`

**Subsystem / component:** File access coordination across readers, writers, locking behavior, SWMR usage, and deployment-specific sharing semantics.

**Hazard family:** H2 Hybrid divergence, H3 Unsafe durability signaling, H8 Operational, privacy, or supply-chain exposure

## Preconditions

- A file is opened by more than one process, node, or tool at overlapping times.
- The writer is modifying the file outside SWMR or another explicitly safe coordination mechanism.
- Locking, publication, or coordination assumptions are weak, disabled, or inconsistent with the target filesystem behavior.

## Accident chain

### Trigger

A reader opens or rereads a file while a writer is actively modifying metadata or raw data without the required coordination model.

### Unsafe state

Readers observe partially updated structures, stale metadata assumptions, or a mix of old and new state that has no safe reconciliation rule.

### Loss / impact

The system can return wrong results, fail to open, or propagate corruption if another tool writes back based on an inconsistent view.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** Unsafe concurrent access can corrupt shared file state, produce incorrect scientific results, or create follow-on writes based on invalid assumptions.

### Likelihood

**3 - Possible.** Multi-process and multi-node access patterns are common, but the hazard depends on deployment choices and whether safe coordination mechanisms are bypassed.

**Risk:** **15 (High)**

## Detectability

Low to moderate. Some failures surface as open errors or reader complaints, but stale reads and inconsistent interpretations can remain plausible unless integration tests or runtime coordination checks catch them.

## Unsafe control actions

- **Not provided when needed:** The system does not enforce SWMR, external locking, or another safe publication mechanism before allowing overlapping access.
- **Provided when unsafe:** A reader is allowed to open or trust a file state while a writer is still publishing non-atomic updates.
- **Wrong timing/order:** Metadata visibility reaches readers before the writer has completed the full update sequence required for a coherent shared view.
- **Too long/too short:** Coordination or lease state expires too early, or readers continue using cached assumptions too long after the file state has changed.

## Safety constraints

- SC-1: The system shall not permit overlapping reader or writer access unless the active coordination model guarantees a coherent file view.
- SC-2: Shared-file deployments shall make the intended access model explicit, including SWMR requirements, locking behavior, and publication rules.
- SC-3: Readers shall fail closed or retry safely when coordination guarantees cannot be established.

## Mitigations

### Prevention

Use SWMR where it fits, enforce external locking when SWMR is not available, publish updates via write-then-rename or generation switching, and validate filesystem semantics in staging before deployment.

### Detection

Add concurrency integration tests, detect lock disablement or environment drift, and surface telemetry for reader errors, stale-read symptoms, and coordination failures.

### Recovery / containment

Treat files touched under unsafe coordination as suspect, isolate them from downstream write-back workflows, and fall back to a known-good published generation when concurrent access assumptions are violated.

## Tests / evidence

- Crash injection: Interrupt a writer during publication and verify that concurrent readers do not accept partially updated state as committed.
- Malformed-input or fuzz coverage: Exercise reopen and read paths with files captured mid-update or with stale coordination metadata.
- Concurrency coverage: Run multi-process and multi-node tests with readers and writers across supported filesystems, with and without SWMR and locking enabled.
- Runtime validation / telemetry: Record lock failures, unexpected overlapping access, reader retry behavior, and stale-read or reopen anomalies.
