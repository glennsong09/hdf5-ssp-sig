# HAZ-016: Chunk cache mis-sizing (`rdcc_nbytes`, `rdcc_nslots`)

**HDF5 SSP vulnerability tags:** `LIB|OPS`

**Subsystem / component:** Raw data chunk cache sizing, workload-specific cache presets, and application-level performance tuning.

**Hazard family:** H8 Operational, privacy, or supply-chain exposure

## Preconditions

- Applications rely on the raw data chunk cache for performance.
- Cache sizing is tuned ad hoc rather than through validated workload classes.
- Performance-sensitive queries or slices are sensitive to cache hit ratio and eviction behavior.

## Accident chain

### Trigger

`rdcc_nbytes` or `rdcc_nslots` is set to values that do not match the workload and may even perform worse than smaller settings.

### Unsafe state

The cache policy amplifies misses, collisions, or memory overhead in a way that makes runtime behavior unpredictable.

### Loss / impact

Performance regresses unexpectedly, tuning becomes fragile, and operators cannot reliably predict how workload changes will behave.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**2 - Minor.** The main consequence is unstable or degraded performance rather than direct correctness loss.

### Likelihood

**3 - Possible.** Cache mis-sizing is a realistic outcome when tuning is based on intuition or copied settings instead of workload evidence.

**Risk:** **6 (Moderate)**

## Detectability

Moderate to high. Benchmarks and cache metrics can reveal the issue, but teams often discover it only after unexplained regressions appear in production.

## Unsafe control actions

- **Not provided when needed:** The system does not offer validated cache presets or benchmark guidance for common workload classes.
- **Provided when unsafe:** Ad hoc cache settings are applied broadly without confirming they improve the target access pattern.
- **Wrong timing/order:** Cache tuning changes are rolled out before representative regression tests are run.
- **Too long/too short:** Oversized or poorly slotted caches persist long enough to create sustained regression or memory pressure.

## Safety constraints

- SC-1: Chunk-cache tuning shall be based on representative workload measurements, not one-off intuition.
- SC-2: Production configurations shall expose only approved cache presets or managed tuning ranges.
- SC-3: Performance-sensitive workflows shall include regression checks for key queries and slice patterns when cache settings change.

## Mitigations

### Prevention

Provide known-good cache presets, expose cache sizing as managed configuration, and require benchmark evidence before adopting custom values.

### Detection

Add performance regression tests for key queries and slices, monitor cache hit behavior, and track tuning changes alongside workload outcomes.

### Recovery / containment

Roll back to known-good presets, isolate risky experiments from production paths, and retune with representative traces rather than reactive ad hoc edits.

## Tests / evidence

- Crash injection: Verify that abrupt termination during cache-heavy workloads does not hide performance-path failures or produce misleading tuning conclusions.
- Malformed-input or fuzz coverage: Exercise cache configuration parsing and boundary handling with invalid or extreme values.
- Concurrency coverage: Benchmark cache-sensitive workloads under concurrent readers and mixed access patterns to detect contention-amplified regressions.
- Runtime validation / telemetry: Track cache settings, hit rates, eviction behavior, memory use, and performance outcomes by workload class.
