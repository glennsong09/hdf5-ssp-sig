# HAZ-010: Chunk size too small

**HDF5 SSP vulnerability tags:** `FMT|LIB|OPS`

**Subsystem / component:** Dataset chunk layout, metadata overhead, cache behavior, and workload-specific access patterns.

**Hazard family:** H8 Operational, privacy, or supply-chain exposure

## Preconditions

- Datasets are chunked.
- Chunk sizes are chosen without workload profiling or organization-wide layout guidance.
- Applications issue many reads or writes whose overhead is sensitive to metadata and cache amplification.

## Accident chain

### Trigger

Datasets are created with chunk sizes that are too small for the actual access pattern and data volume.

### Unsafe state

The file layout multiplies metadata traffic, cache churn, and I/O overhead beyond what the workload can safely tolerate.

### Loss / impact

Performance degrades severely, metadata and storage overhead grow unnecessarily, and time-sensitive processing can miss operational targets.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**3 - Moderate.** The main effect is degraded performance and higher overhead, which can still cause material operational disruption in large-scale workflows.

### Likelihood

**4 - Likely.** Poor chunk sizing is a common schema and application design mistake, especially when defaults or ad hoc values are used.

**Risk:** **12 (High)**

## Detectability

Moderate to high. Benchmarking and profiling expose the issue, but teams often discover it only after production workloads grow large enough to trigger the cost.

## Unsafe control actions

- **Not provided when needed:** Dataset creation paths do not validate chunk sizes against expected access patterns or policy guidance.
- **Provided when unsafe:** Small chunks are accepted for workloads that need larger, more contiguous I/O units.
- **Wrong timing/order:** Layout decisions are finalized before profiling or representative access analysis is complete.
- **Too long/too short:** Chunk dimensions are so small that metadata and scheduling overhead dominate useful data transfer.

## Safety constraints

- SC-1: Chunked dataset definitions shall be validated against representative workload access patterns before production use.
- SC-2: Schema and ingestion tooling shall reject chunk configurations known to create pathological metadata or cache amplification.
- SC-3: Performance-sensitive deployments shall maintain approved chunk-size guidance for common workload classes.

## Mitigations

### Prevention

Define chunking guidelines, profile common slicing patterns, and fail fast on pathological chunk configurations in schema validation or ingest tooling.

### Detection

Benchmark representative reads and writes, monitor metadata overhead and cache churn, and flag datasets whose chunk counts or request patterns exceed expected bounds.

### Recovery / containment

Repack or regenerate datasets with corrected chunk layouts, isolate known-bad layouts from latency-sensitive paths, and document migration plans for affected schemas.

## Tests / evidence

- Crash injection: Verify that severe chunk-count amplification does not leave recovery or maintenance tooling unable to complete safely under interruption.
- Malformed-input or fuzz coverage: Exercise layout validation with extreme chunk configurations to ensure early rejection of pathological settings.
- Concurrency coverage: Benchmark multi-reader and mixed read/write workloads to measure how small chunks amplify contention and cache churn.
- Runtime validation / telemetry: Track chunk counts, metadata I/O rates, cache hit ratios, and end-to-end latency for representative datasets.
