# HAZ-013: High-level API thread-safety gaps

**HDF5 SSP vulnerability tags:** `LIB|TCD|OPS`

**Subsystem / component:** High-level HDF5 APIs, language bindings, wrapper libraries, and build-matrix compatibility under threaded execution.

**Hazard family:** H5 In-memory structural corruption, H7 Extension boundary violation, H8 Operational, privacy, or supply-chain exposure

## Preconditions

- Applications use thread-safe builds or settings together with high-level libraries and bindings.
- The supported threading model differs across bindings, versions, or build flags.
- The deployment lacks a tested compatibility matrix for the chosen combinations.

## Accident chain

### Trigger

Threaded code mixes HDF5 core and high-level APIs or bindings in a configuration whose thread-safety guarantees do not actually align.

### Unsafe state

The process executes concurrent operations through wrappers or high-level code paths that violate the assumptions of the loaded build combination.

### Loss / impact

Crashes, undefined behavior, or inconsistent results occur, and the failure may appear only under production-level concurrency.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**3 - Moderate.** Failures can crash processes or create undefined behavior, but they are usually scoped to runtime stability rather than durable file corruption.

### Likelihood

**3 - Possible.** The issue depends on build and binding combinations, yet mixed-language and wrapper-heavy deployments make unsupported combinations realistic.

**Risk:** **9 (Moderate)**

## Detectability

Moderate. Stress testing and sanitizers can expose the problem, but many combinations appear stable until concurrency increases or specific interleavings occur.

## Unsafe control actions

- **Not provided when needed:** The project does not define or test a supported matrix of HDF5 flags, wrapper versions, and threading models.
- **Provided when unsafe:** High-level APIs or bindings are used concurrently in combinations whose thread-safety semantics are undocumented or unsupported.
- **Wrong timing/order:** Concurrent calls cross binding or API boundaries before shared state and initialization paths are known to be safe.
- **Too long/too short:** A wrapper retains shared state or assumptions longer than the underlying build can safely support under threaded access.

## Safety constraints

- SC-1: Supported threading models shall be documented for each language binding and high-level API layer.
- SC-2: Build and dependency combinations used in production shall be validated under representative threaded workloads before release.
- SC-3: Unsupported or untested concurrency combinations shall fail fast or be blocked by build and deployment guardrails.

## Mitigations

### Prevention

Standardize a tested build matrix, document supported binding versions and threading modes, and keep concurrency-sensitive usage patterns out of unsupported high-level paths.

### Detection

Run concurrency stress tests in CI, use sanitizers where possible, and track crashes or race-like failures by build and binding combination.

### Recovery / containment

Pin to known-good combinations, disable threaded access to unsafe bindings, and isolate concurrency-sensitive work into supported lower-level or process-isolated paths.

## Tests / evidence

- Crash injection: Interrupt and restart threaded workloads around high-level API operations to verify that failure handling does not mask thread-safety defects.
- Malformed-input or fuzz coverage: Exercise error-heavy high-level code paths under concurrency to expose unsafe shared-state assumptions.
- Concurrency coverage: Run threaded stress tests across the supported build matrix, bindings, and sanitizer configurations.
- Runtime validation / telemetry: Record crashes, race-detector findings, build flags, binding versions, and concurrency-mode selection in production diagnostics.
