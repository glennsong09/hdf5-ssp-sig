# HAZ-001: Crash during metadata update

**HDF5 SSP vulnerability tags:** `FMT|LIB|OPS`

**Subsystem / component:** Metadata update path spanning metadata cache, object headers, B-tree / heap handling, and the durability boundary exposed through flush / close.

**Hazard family:** H1 On-disk metadata inconsistency, H2 Hybrid divergence, H3 Unsafe durability signaling

## Preconditions

- The library is updating metadata that will reference a new or changed on-disk block.
- The referenced block, related raw data, or dependent metadata is still in flight or only resident in memory.
- The environment does not provide stronger ordering or transactional guarantees than HDF5 explicitly enforces.

## Accident chain

### Trigger

Crash, power loss, process abort, or `SIGKILL` occurs after a metadata pointer is updated but before the referenced bytes are durably persisted.

### Unsafe state

Durable metadata points to incomplete, stale, or uninitialized bytes, and in-memory state that could have repaired or completed the update is lost.

### Loss / impact

The file may fail to open, may open with silent corruption, or may expose a durable state that overstates what was actually committed.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** A broken metadata reference can make the file unreadable, hide valid data behind inconsistent structure, or produce silent corruption after reopen.

### Likelihood

**4 - Likely.** Crash and interruption events are common enough in real deployments, and this hazard is reachable whenever metadata ordering depends on incomplete durability guarantees.

**Risk:** **20 (Very High)**

## Detectability

Moderate. Reopen validation, checksums, metadata consistency checks, and crash telemetry can detect many failures, but some outcomes can still appear as plausible reads until deeper validation occurs.

## Unsafe control actions

- **Not provided when needed:** The library does not enforce a required ordering step that ensures referenced bytes are durable before publishing the metadata pointer.
- **Provided when unsafe:** `flush`, `close`, or equivalent success feedback is surfaced while the referenced block is not yet safely persisted.
- **Wrong timing/order:** Metadata is written or made durable before the data or subordinate structure it references.
- **Too long/too short:** A partial write or interrupted persistence window leaves only part of the intended metadata update durable.

## Safety constraints

- SC-1: The library shall not make a metadata pointer durable until the referenced bytes and dependent structures satisfy the required durability ordering.
- SC-2: The library shall not report `flush`, `close`, or equivalent success in a way that overstates on-disk safety at the current durability boundary.
- SC-3: Reopen paths shall detect and fail closed on incomplete or stale metadata references rather than accepting them as valid state.

## Mitigations

### Prevention

Enforce metadata write-ordering constraints, stage updates so new references are published only after dependent bytes are durable, and use checkpointing or journaling-style mechanisms where appropriate.

### Detection

Run stronger metadata validation on reopen, use checksums or integrity checks where available, and emit telemetry for interrupted writes, failed flushes, and recovery-triggering conditions.

### Recovery / containment

Preserve a last-known-good checkpoint or generation, fail closed on detected inconsistency, and provide repair or salvage tooling that can identify a safe recovery point.

## Tests / evidence

- Crash injection: Interrupt execution at multiple points between metadata pointer update and durability completion, then verify reopen behavior and file integrity.
- Malformed-input or fuzz coverage: Exercise reopen and parser logic with files that contain dangling, stale, or partially persisted metadata references.
- Concurrency coverage: Combine interruption scenarios with concurrent readers or reopen attempts to verify that partially updated metadata is not treated as committed state.
- Runtime validation / telemetry: Record flush / close failures, restart-after-write events, integrity-check failures, and reopen validation outcomes.
