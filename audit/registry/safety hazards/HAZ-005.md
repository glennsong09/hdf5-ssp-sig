# HAZ-005: Out-of-space (ENOSPC) during write or close

**HDF5 SSP vulnerability tags:** `OPS|LIB|FMT`

**Subsystem / component:** Write, flush, and close paths spanning metadata updates, raw data writes, free-space management, and the VFD / filesystem boundary.

**Hazard family:** H1 On-disk metadata inconsistency, H2 Hybrid divergence, H3 Unsafe durability signaling, H8 Operational, privacy, or supply-chain exposure

## Preconditions

- The file or backing filesystem is near capacity.
- The application is performing writes, flushes, or close-time metadata work that still requires free space.
- No reserved headroom, preflight capacity check, or generation-switch plan is in place.

## Accident chain

### Trigger

The filesystem returns `ENOSPC` during a write, flush, or close sequence.

### Unsafe state

Only part of the intended update becomes durable, close semantics are incomplete, and the file may remain in an ambiguous or inconsistent state.

### Loss / impact

Writes fail, close itself can fail, the file can become corrupted or unusable, and operations may need to stop or switch targets immediately.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** A space exhaustion event during persistence can cause data loss, metadata inconsistency, and prolonged downtime if the file cannot be safely closed or recovered.

### Likelihood

**3 - Possible.** Capacity incidents are common enough in production systems, but the hazard depends on workload growth, provisioning, and monitoring discipline.

**Risk:** **15 (High)**

## Detectability

Moderate. `ENOSPC` is usually surfaced immediately, but the full integrity impact may not be known until reopen, validation, or later read attempts.

## Unsafe control actions

- **Not provided when needed:** The system does not check or reserve sufficient capacity before starting a write sequence that must complete atomically enough to remain safe.
- **Provided when unsafe:** The application continues writing to or trusting a file after `ENOSPC` has already invalidated safe completion assumptions.
- **Wrong timing/order:** Space is consumed by dependent metadata or close-time work after the system has already committed to a layout that requires additional free space.
- **Too long/too short:** Cleanup or rollback after `ENOSPC` stops too early, leaving a partially updated file still treated as usable.

## Safety constraints

- SC-1: The system shall treat `ENOSPC` during write, flush, or close as a file-safety event, not just a transient I/O error.
- SC-2: Write paths shall define and enforce headroom or preflight capacity checks for operations that could leave the file unsafe if interrupted by space exhaustion.
- SC-3: After `ENOSPC`, the file shall be marked suspect until integrity validation or controlled recovery confirms a safe state.

## Mitigations

### Prevention

Enforce free-space headroom, preflight capacity before large writes, reserve a safety margin, and prefer generation switching or new output targets when capacity is uncertain.

### Detection

Monitor free-space trends, alert on low-capacity thresholds, and run integrity validation after any write or close failure caused by `ENOSPC`.

### Recovery / containment

Treat the current file as failed, switch to a new file generation or target location, preserve the suspect artifact for validation, and recover from backups or replayable inputs when needed.

## Tests / evidence

- Crash injection: Combine interruption tests with simulated `ENOSPC` at multiple write and close points to verify fail-safe behavior.
- Malformed-input or fuzz coverage: Exercise reopen and validation logic against files left partially updated after space-exhaustion failures.
- Concurrency coverage: Verify that readers and other writers do not continue using a file that has entered a suspect state after `ENOSPC`.
- Runtime validation / telemetry: Record `ENOSPC` events, close failures, integrity-check results, and automatic generation-switch actions.
