# HAZ-008: File locking pitfalls or disabling via environment variable

**HDF5 SSP vulnerability tags:** `OPS|LIB`

**Subsystem / component:** File locking behavior, deployment configuration, environment variables such as `HDF5_USE_FILE_LOCKING`, and shared-file access control.

**Hazard family:** H2 Hybrid divergence, H8 Operational, privacy, or supply-chain exposure

## Preconditions

- Multiple processes or tools may access the same file concurrently.
- File locking is relied on as part of the coordination model, or users believe it is active.
- Environment or platform configuration can disable, weaken, or bypass locking behavior.

## Accident chain

### Trigger

Locking is disabled, misconfigured, or behaves differently than operators expect, including through environment-variable overrides.

### Unsafe state

Processes interact with the same file under false coordination assumptions, allowing overlapping reads or writes that violate safe access rules.

### Loss / impact

Concurrent access hazards increase, corruption incidents become more likely, and production behavior can drift from test assumptions.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**4 - Major.** A disabled or mismatched locking model can silently reopen corruption and stale-read hazards that operators think are already controlled.

### Likelihood

**3 - Possible.** Environment drift and filesystem-specific locking behavior are common enough to make this a recurring deployment risk.

**Risk:** **12 (High)**

## Detectability

Low to moderate. Lock failures or warnings may appear, but silent overrides and environment drift can leave the system operating unsafely without immediate symptoms.

## Unsafe control actions

- **Not provided when needed:** The deployment does not enforce or verify the locking mode required for its shared-file access model.
- **Provided when unsafe:** Locking is disabled through environment configuration while concurrent access patterns still assume it is active.
- **Wrong timing/order:** Readers or writers begin access before lock state has been confirmed, or proceed after a lock failure without a safe fallback.
- **Too long/too short:** A lock is released too early or not held for the full interval required to maintain safe access.

## Safety constraints

- SC-1: Production deployments shall make file-locking configuration explicit and resistant to unintended environment overrides.
- SC-2: Workflows that require locking shall fail closed when the expected lock state cannot be established or verified.
- SC-3: Shared-file tests shall cover the actual locking behavior of the target filesystems and runtime environments.

## Mitigations

### Prevention

Standardize environment settings, prevent unsafe `HDF5_USE_FILE_LOCKING` overrides, and define a replacement coordination model if locking must be disabled.

### Detection

Test locking behavior on target filesystems, surface lock configuration at startup, and alert on lock failures or unexpected disablement.

### Recovery / containment

Quarantine files touched under uncertain lock conditions, stop overlapping access, and republish from a verified safe state before resuming shared workflows.

## Tests / evidence

- Crash injection: Verify that lock loss or process interruption does not leave the system assuming safe coordination when it no longer exists.
- Malformed-input or fuzz coverage: Exercise file-open paths with inconsistent or missing lock metadata and confirm fail-safe behavior.
- Concurrency coverage: Run negative and positive locking tests across supported filesystems, including scenarios with environment overrides enabled.
- Runtime validation / telemetry: Record effective lock mode, lock acquisition failures, override detection, and overlapping access attempts.
