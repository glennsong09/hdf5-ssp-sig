# HAZ-015: Misuse of metadata-cache flush controls

**HDF5 SSP vulnerability tags:** `LIB|OPS|FMT`

**Subsystem / component:** Metadata cache flush-control APIs such as `H5Odisable_mdc_flushes`, transaction-like update patterns, and centralized metadata lifecycle management.

**Hazard family:** H1 On-disk metadata inconsistency, H2 Hybrid divergence, H3 Unsafe durability signaling

## Preconditions

- Applications use metadata-cache flush-control APIs.
- The calling code assumes these controls can safely approximate transaction behavior.
- Usage is scattered or insufficiently constrained by invariants and tests.

## Accident chain

### Trigger

Flush-control APIs are used incorrectly, leaving required metadata updates deferred or blocked beyond the safe window.

### Unsafe state

Metadata required for reachability or consistency remains non-flushable while other state advances, creating an unsafe mismatch between memory and durable structure.

### Loss / impact

Data becomes inaccessible, files can be corrupted, and operators may believe updates are safe even though critical metadata was intentionally withheld from flush.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**4 - Major.** Misusing flush controls can directly affect metadata integrity and make valid data unreachable or unsafe after interruption.

### Likelihood

**2 - Uncommon.** The APIs are specialized and explicitly warned about, but misuse remains plausible when applications attempt transaction-like patterns without tight discipline.

**Risk:** **8 (Moderate)**

## Detectability

Low to moderate. Problems may not appear until reopen, crash recovery, or a code path assumes metadata became durable when it did not.

## Unsafe control actions

- **Not provided when needed:** The application does not enforce centralized invariants and tests around flush-control API use.
- **Provided when unsafe:** Metadata flushes are disabled in code paths that cannot guarantee safe completion or controlled re-enable behavior.
- **Wrong timing/order:** Flushes are re-enabled after dependent state has already been published or assumed durable.
- **Too long/too short:** Flush suppression remains active too long, allowing deferred metadata to accumulate beyond a safe transaction-like window.

## Safety constraints

- SC-1: Flush-control APIs shall only be used inside a tested, centralized pattern with explicit entry, exit, and rollback rules.
- SC-2: The system shall not expose or publish state that depends on metadata still blocked from flush.
- SC-3: Re-enable and validation steps shall complete before success is reported to callers or downstream workflows.

## Mitigations

### Prevention

Use these APIs only with a tested transaction-like pattern, centralize the implementation, and add invariants that confirm all required metadata can flush again before success.

### Detection

Add targeted tests around disable/enable sequences, reopen validation, and telemetry for long-lived flush suppression or failed re-enable operations.

### Recovery / containment

Fail closed when flush state is uncertain, keep suspect files out of downstream use, and recover from a verified prior state if metadata accessibility cannot be restored safely.

## Tests / evidence

- Crash injection: Interrupt execution while metadata flushes are disabled and verify reopen behavior after each critical sequence point.
- Malformed-input or fuzz coverage: Exercise reopen and validation logic on files created from incomplete or misordered flush-control sequences.
- Concurrency coverage: Verify that concurrent operations cannot observe or act on state that depends on still-suppressed metadata flushes.
- Runtime validation / telemetry: Record flush-disable intervals, re-enable outcomes, metadata validation failures, and uses of restricted APIs in production.
