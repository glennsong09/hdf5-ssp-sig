# HAZ-003: Extension boundary violates core invariants

**HDF5 SSP vulnerability tags:** `EXT|TCD|SCD`

**Subsystem / component:** Filter, VOL, VFD, wrapper, and plugin boundaries where extension-provided data or metadata is accepted by the core library.

**Hazard family:** H7 Extension boundary violation, H5 In-memory structural corruption, H8 Operational, privacy, or supply-chain exposure

## Preconditions

- An extension, plugin, wrapper, or dependency participates in I/O, metadata handling, or object lifecycle management.
- The core library relies on the extension to respect address, size, ordering, or lifecycle invariants.
- Extension trust, provenance, or runtime isolation is limited or absent.

## Accident chain

### Trigger

Buggy, incompatible, or untrusted extension code returns data or metadata that violates core address, size, or lifecycle expectations.

### Unsafe state

The core library accepts unsafe extension output as valid internal or on-disk state, allowing corruption or policy violations to cross the boundary unchecked.

### Loss / impact

The result can include wrong writes, crashes, privacy leakage, unsafe code loading outcomes, or downstream corruption that appears to originate in the core library.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** A violated extension boundary can compromise core invariants, produce persistent corruption, or introduce unsafe behavior through trusted execution paths.

### Likelihood

**3 - Possible.** The hazard depends on extension use, but plugin-rich deployments, wrappers, and external dependencies make boundary failures realistic.

**Risk:** **15 (High)**

## Detectability

Moderate. Boundary violations can be found by invariant checks, loader controls, and targeted testing, but subtle semantic drift or dependency issues may look valid until they trigger corruption later.

## Unsafe control actions

- **Not provided when needed:** The core library does not validate extension outputs against required invariants before accepting them.
- **Provided when unsafe:** Untrusted or incompatible extensions are loaded or invoked inside the trusted process boundary.
- **Wrong timing/order:** Extension outputs are consumed before version, lifecycle, or consistency checks complete.
- **Too long/too short:** Extension state persists beyond its safe lifetime, or partial boundary validation leaves only some invariants enforced.

## Safety constraints

- SC-1: The core library shall validate extension-provided addresses, sizes, object lifecycles, and state transitions before treating them as trusted input.
- SC-2: Extension loading shall be restricted to trusted, compatible, and explicitly allowed artifacts for the deployment.
- SC-3: High-risk extensions shall have a containment option, such as sandboxing, process isolation, or strict allowlisting.

## Mitigations

### Prevention

Use allowlists, signed or provenance-verified artifacts, compatibility checks, explicit trust configuration, and strict invariant validation at every extension boundary.

### Detection

Add boundary-focused tests, assertions, and telemetry for invariant violations, plugin load events, dependency drift, and unexpected extension behavior.

### Recovery / containment

Disable or quarantine suspect extensions, fail closed on invalid boundary outputs, and isolate high-risk plugins from the main process or production data paths.

## Tests / evidence

- Crash injection: Force extension failures or abrupt termination during boundary interactions and verify that unsafe partial outputs are not accepted as valid state.
- Malformed-input or fuzz coverage: Fuzz extension outputs and plugin-provided metadata to ensure the core rejects invalid addresses, sizes, and lifecycle transitions.
- Concurrency coverage: Exercise extension callbacks and shared state under concurrent access to verify that timing-sensitive invariant violations are detected.
- Runtime validation / telemetry: Log plugin load decisions, invariant check failures, version mismatches, and boundary rejection events.
