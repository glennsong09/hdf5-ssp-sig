# HAZ-014: Portability hazard from missing filter plugins

**HDF5 SSP vulnerability tags:** `EXT|OPS`

**Subsystem / component:** Filter plugins, dataset decoding requirements, packaging, and cross-environment portability of compressed or transformed data.

**Hazard family:** H7 Extension boundary violation, H8 Operational, privacy, or supply-chain exposure

## Preconditions

- Files depend on non-core filter plugins for decoding.
- Data is shared across systems, containers, clusters, or partners with different plugin availability.
- Packaging or data contracts do not guarantee decoder presence everywhere the file may be read.

## Accident chain

### Trigger

A file is opened on a system that lacks one or more required filter plugins.

### Unsafe state

The file format is syntactically present, but required decode behavior is unavailable in the current environment.

### Loss / impact

Datasets become unreadable, portability assumptions fail, and operational workflows stall or silently diverge across environments.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**3 - Moderate.** Missing plugins usually affect availability and portability rather than integrity, but they can still block critical reads and processing pipelines.

### Likelihood

**3 - Possible.** Cross-environment data movement is common, and plugin packaging drift is a realistic source of incompatibility.

**Risk:** **9 (Moderate)**

## Detectability

High on open or decode, but often late. The absence of a plugin is usually explicit when the dataset is accessed, yet teams often find out only after distribution.

## Unsafe control actions

- **Not provided when needed:** The system does not validate decoder availability before ingest, publication, or cross-environment transfer.
- **Provided when unsafe:** Files are published with plugin-dependent encodings into environments that have no guaranteed plugin distribution path.
- **Wrong timing/order:** Data contracts or metadata describing filter requirements are added too late or omitted until after the file has already been shared.
- **Too long/too short:** Plugin dependencies persist longer than the deployment or packaging strategy that originally supplied them.

## Safety constraints

- SC-1: Files using non-core filters shall declare their decoder requirements in a way that deployment and ingest tooling can verify.
- SC-2: Publication workflows shall not ship plugin-dependent datasets into environments that lack an approved decoder distribution path.
- SC-3: Portability-sensitive data products shall prefer standardized or bundled filter support over ad hoc plugin dependencies.

## Mitigations

### Prevention

Standardize approved filters, bundle required plugins in deployed apps or containers, validate decoder availability at ingest, and record filter requirements in metadata and contracts.

### Detection

Add portability checks in CI and release pipelines, verify plugin presence at startup, and flag files whose filter dependencies are not satisfied in the target environment.

### Recovery / containment

Provide compatible decoding environments, republish data with supported filters when needed, and prevent downstream workflows from assuming readability until plugin requirements are met.

## Tests / evidence

- Crash injection: Verify that failures while loading or invoking filter plugins do not leave partially processed outputs treated as valid results.
- Malformed-input or fuzz coverage: Exercise filter metadata and plugin-discovery paths with missing, mismatched, or invalid filter declarations.
- Concurrency coverage: Test multi-process reads where plugin discovery and loading occur concurrently across shared environments.
- Runtime validation / telemetry: Record required filter IDs, plugin load outcomes, decode failures, and cross-environment portability checks.
