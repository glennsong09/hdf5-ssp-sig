# ATK-001: Malicious metadata size and address fields corrupt parser memory

**CASSE:** Data • Poisoning • Core

**HDF5 SSP vulnerability tags:** `FMT|LIB`

## Preconditions

- An application opens or inspects an HDF5 file from an untrusted or cross-tenant source.
- The file reaches low-level metadata decode paths that interpret lengths, offsets, addresses, or counts.

## Trigger / entry point

Open, metadata cache reconstruction, or object traversal processes crafted address-length fields that drive allocation and copy sizes.

## Vulnerability mechanism (CWE-style)

Unchecked or inconsistently validated metadata lengths and addresses lead to integer wraparound, undersized allocations, and heap-based out-of-bounds writes or reads. Relevant CWEs include `CWE-122`, `CWE-787`, `CWE-125`, and `CWE-190`.

## Expected impact

Process crash, corrupted in-memory metadata state, denial of service, and credible code-execution potential in the parser process.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** These bugs sit in core metadata decoding and can directly corrupt heap state before the library has established a trusted view of the file.

### Likelihood

**5 - Almost certain.** The CVE set shows repeated regressions in the same class across multiple releases and parser paths, and exploitation only requires delivering a malformed file.

**Risk:** **25 (Critical)**

## Exploitability notes

Many CVE records describe the attack as "local," but under the HDF5 threat model this is still a high-risk untrusted-data attack because library consumers routinely parse attacker-supplied files on local hosts, shared services, and batch nodes.

## Detection

Targeted fuzzing of superblock, object-header, and cache-image decode paths; ASan/UBSan builds; crash deduplication keyed to metadata decode functions; and preflight validation that rejects impossible lengths, offsets, and object counts before allocation-heavy code runs.

## Mitigations

Centralize bounds checks for all metadata address-length decoders, use overflow-safe arithmetic helpers, reject impossible structural combinations early, and add fail-closed validation before cache image or object header reconstruction proceeds.

## Tests / evidence

- Regression coverage for `H5F_addr_decode_len`, `H5O__linfo_decode`, and related malformed-address paths.
- Fuzz seeds with oversized lengths, truncated metadata records, and conflicting offsets.
- CVE evidence from `CVE-2025-6516`, `CVE-2024-32620`, `CVE-2021-45830`, and `CVE-2024-29166`.
