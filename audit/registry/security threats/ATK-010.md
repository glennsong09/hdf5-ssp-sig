# ATK-010: Malformed metadata can trigger null or uninitialized pointer dereference

**CASSE:** Data • Poisoning • Core

**HDF5 SSP vulnerability tags:** `FMT|LIB`

## Preconditions

- Untrusted metadata reaches parser states that assume optional structures, decoded pointers, or cache entries are already valid.
- The application processes the file in a context where a crash is still a meaningful availability or integrity event.

## Trigger / entry point

Malformed file structures drive error handling or cache/attribute cleanup paths into null or uninitialized pointer dereference.

## Vulnerability mechanism (CWE-style)

The parser uses pointers before validation or after incomplete initialization, usually during cleanup, serialization, or decode fallback logic. Relevant CWEs include `CWE-476` and `CWE-457`.

## Expected impact

Immediate denial of service, inconsistent cleanup behavior, and secondary integrity risk when the crash occurs during metadata flush or close.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**3 - Moderate.** These bugs usually crash the process rather than provide a strong execution primitive, but they still break availability and can interrupt metadata updates.

### Likelihood

**4 - Likely.** Null or uninitialized pointer dereferences are comparatively easy to trigger once malformed metadata reaches an under-checked cleanup or decode path.

**Risk:** **12 (High)**

## Exploitability notes

Even when exploitation stops at a crash, this class matters for automated pipelines because repeated attacker submissions can keep workers unavailable or interrupt file-rewrite workflows mid-operation.

## Detection

UBSan and MSan builds, negative tests that force partial initialization and cleanup, and crash triage that distinguishes parser-state bugs from generic open failures.

## Mitigations

Require explicit pointer validity checks before use, initialize all parser state deterministically, and simplify cleanup logic so it does not assume earlier decode stages succeeded.

## Tests / evidence

- Regression tests for `H5C__flush_single_entry`, `H5O__cache_chk_serialize`, `H5O__dtype_decode_helper`, and `H5A__attr_release_table`.
- Fault-injection tests that simulate partial decode and cleanup.
- CVE evidence from `CVE-2025-6858`, `CVE-2025-2926`, `CVE-2021-46243`, and `CVE-2024-32611`.
