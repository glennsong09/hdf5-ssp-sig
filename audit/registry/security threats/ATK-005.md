# ATK-005: Datatype decoding and conversion logic corrupts memory

**CASSE:** Data â€˘ Poisoning â€˘ Core

**HDF5 SSP vulnerability tags:** `FMT|LIB`

## Preconditions

- The attacker can supply datasets or attributes with crafted datatypes, references, or compound layouts.
- The application opens, inspects, converts, or copies those datatypes using normal library APIs.

## Trigger / entry point

Datatype metadata is decoded, copied, reopened, normalized, or converted during read or write preparation.

## Vulnerability mechanism (CWE-style)

Bit offsets, element widths, reference layouts, and derived native-type sizes are trusted without complete validation, causing out-of-bounds memory operations during datatype handling. Relevant CWEs include `CWE-122`, `CWE-787`, `CWE-120`, and `CWE-190`.

## Expected impact

Heap corruption, wrong data conversion results, denial of service, and plausible code execution in applications that process untrusted scientific data.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** Datatype logic shapes almost every higher-level operation in HDF5, so corruption here has broad reach and can also silently damage data interpretation.

### Likelihood

**4 - Likely.** The vulnerable paths are not as universal as basic metadata decode, but the CVE history spans several independent datatype subsystems and multiple release lines.

**Risk:** **20 (Very High)**

## Exploitability notes

This class is dangerous because even files that pass superficial metadata checks can still trigger memory corruption later when the application requests native conversion, compound-member access, or reference handling.

## Detection

Datatype-focused fuzzing, property-based tests over extreme bit widths and nested compound types, and sanitizer-backed regression tests for every type conversion helper touched by past CVEs.

## Mitigations

Define explicit invariants for datatype width, offset, and nesting depth; use checked arithmetic for every derived size; and require negative tests for malformed datatype graphs before new datatype features ship.

## Tests / evidence

- Regression tests for `H5T__bit_copy`, `H5T__conv_struct_opt`, `H5T_copy_reopen`, `H5T__get_native_type`, `H5T__bit_find`, `H5T__ref_mem_setnull`, and `H5O__dtype_encode_helper`.
- Fuzz cases for malformed compound, reference, and variable-width datatype metadata.
- CVE evidence from `CVE-2025-2309`, `CVE-2024-33877`, `CVE-2024-32619`, `CVE-2024-32618`, `CVE-2024-29163`, `CVE-2024-32624`, `CVE-2024-32610`, and `CVE-2024-32616`.
