# ATK-011: Malicious plugin resolution executes untrusted filter, VOL, or VFD code

**CASSE:** Data • Poisoning • External

**HDF5 SSP vulnerability tags:** `EXT|SCD|OPS`

## Preconditions

- The deployment allows dynamic loading of filters, VOL connectors, or VFDs.
- Plugin discovery is influenced by file metadata, environment variables, search paths, or writable plugin directories.

## Trigger / entry point

An attacker supplies an HDF5 file that references a plugin ID or workflow requirement that causes the process to resolve and load external code.

## Vulnerability mechanism (CWE-style)

The library or surrounding workflow accepts executable extension code from an uncontrolled search path or insufficiently trusted registry. Relevant CWEs include `CWE-427` and `CWE-494`.

## Expected impact

Arbitrary code execution in the parsing process, unauthorized filesystem or network access through the loaded plugin, and compromise of downstream jobs that trust the same runtime environment.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** Once an untrusted plugin is loaded, the attacker is executing native code inside the application context.

### Likelihood

**4 - Likely.** This requires dynamic-extension support and unsafe discovery policy, but those conditions are realistic in scientific workflows that rely on third-party filters and connectors.

**Risk:** **20 (Very High)**

## Exploitability notes

This is the cleanest path from a malicious file to code execution when extension loading is enabled, because the file only needs to steer the runtime toward attacker-controlled plugin resolution rather than exploit a memory-safety bug.

## Detection

Log every plugin load decision, audit effective plugin search paths at startup, alert on environment-driven path changes, and test whether privileged services ignore untrusted `HDF5_PLUGIN_PATH`-style inputs.

## Mitigations

Disable dynamic plugin loading by default in high-risk contexts, enforce signed and allowlisted plugins, make plugin directories root-owned and non-writable, and run extension code in isolated processes or sandboxes where feasible.

## Tests / evidence

- Path-hijack tests that verify untrusted directories and environment variables are ignored.
- Signature-verification and allowlist tests for approved plugins.
- Integration tests that confirm unknown plugin IDs fail closed instead of probing writable paths.
