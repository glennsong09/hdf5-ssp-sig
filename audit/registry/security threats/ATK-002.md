# ATK-002: Crafted object-header messages overflow metadata serialization paths

**CASSE:** Data • Poisoning • Core

**HDF5 SSP vulnerability tags:** `FMT|LIB`

## Preconditions

- The attacker controls HDF5 metadata that drives object-header, fsinfo, layout, mtime, or cache-message serialization.
- The application performs open, flush, close, repack, or metadata rewrite activity on the file.

## Trigger / entry point

Malformed object-header messages or related metadata are encoded, flushed, or reserialized by core library routines.

## Vulnerability mechanism (CWE-style)

Metadata fields are copied or encoded into fixed or insufficient buffers without proving that the destination size matches the serialized representation. Relevant CWEs include `CWE-122`, `CWE-120`, and `CWE-787`.

## Expected impact

Heap corruption during normal metadata maintenance, leading to crashes, parser compromise, or emission of corrupted output files.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** Object headers are central to file structure, so corruption in these paths can break integrity and can also become code-execution primitives.

### Likelihood

**4 - Likely.** The attack needs specific metadata forms, but the affected code sits on common open/flush/close workflows and has multiple independent CVEs.

**Risk:** **20 (Very High)**

## Exploitability notes

This class is especially dangerous for services that rewrite or normalize uploaded HDF5 files, because a file that appears readable can still trigger the bug later during flush, repack, or close.

## Detection

Negative tests for every object-header message family, sanitizer-backed integration tests that open and then rewrite malformed files, and fuzz coverage that mutates encoded message lengths and chunk boundaries.

## Mitigations

Compute serialized sizes once and assert them before writing, isolate encode/decode logic behind shared checked helpers, and reject malformed object-header messages before they can be re-emitted or flushed.

## Tests / evidence

- Regression tests for `H5O__chunk_protect`, `H5O__fsinfo_encode`, `H5O__mtime_new_encode`, `H5O_msg_flush`, and `H5O__layout_encode`.
- Fuzz cases that mutate message count, chunk size, and encoded payload length.
- CVE evidence from `CVE-2025-6818`, `CVE-2025-6816`, `CVE-2025-6750`, `CVE-2025-2912`, `CVE-2024-33875`, and `CVE-2024-33874`.
