# ATK-012: Pathological chunk layouts amplify I/O and overload storage backends

**CASSE:** Data • Poisoning • Storage

**HDF5 SSP vulnerability tags:** `OPS|FMT`

## Preconditions

- The application reads HDF5 files from a shared filesystem, object store, or networked storage service.
- The workflow does not preflight chunk layout, chunk count, or expected I/O fan-out before full ingest.

## Trigger / entry point

A crafted file uses extremely small chunks, high chunk counts, or storage-hostile layout metadata so normal reads generate excessive tiny I/O operations.

## Vulnerability mechanism (CWE-style)

The file format permits metadata choices that drive disproportionate backend work, and the workflow lacks operational limits or layout policy checks. Relevant CWEs include `CWE-400` and `CWE-770`.

## Expected impact

Storage latency spikes, service degradation for unrelated users, excessive object-store request cost, and failed or stalled ingestion jobs even when the file is otherwise valid.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**

Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**4 - Major.** The main effect is availability and cost impact on shared storage rather than direct parser compromise, but the blast radius can span many tenants and jobs.

### Likelihood

**4 - Likely.** This attack can be mounted with a syntactically valid file and does not require triggering undefined behavior in the library.

**Risk:** **16 (Very High)**

## Exploitability notes

This threat is strongest in cloud and HPC environments where many readers share the same backend and per-request overhead is high relative to payload size.

## Detection

Pre-ingest layout scanning, telemetry on tiny-read rates and per-file I/O amplification, and alerts for files whose chunk geometry implies unreasonable request volume.

## Mitigations

Set minimum chunk-size and maximum chunk-count policies, preflight files before broad distribution, throttle per-job I/O, and isolate high-risk ingest workflows from production storage tiers.

## Tests / evidence

- Policy tests that reject files below the configured minimum chunk size or above maximum chunk count.
- Benchmark tests that measure backend request amplification from hostile layouts.
- Operational runbooks that document throttling, quarantine, and preflight steps before full ingest.
