# EXP-005: Credential handling for ROS3 and tooling can leak secrets

**HDF5 SSP vulnerability tags:** `EXT|TCD|PRV|OPS`

**Exposure family:** P4 Unintended inclusion during processing

**Lifecycle stage:** Processing and conversion

**Sensitive asset:** Access keys, tokens, signed URLs, and other credentials used to reach private object storage.

## Trigger

Credentials are passed through APIs, environment variables, config files, or command-line arguments during ROS3 access, debugging, CI, or helper-tool execution.

## Exposure surface

Process lists, shell history, environment dumps, crash reports, debug logs, CI artifacts, local config files, and copied scripts that preserve secrets longer than intended.

## Disclosure

Leaked credentials allow unauthorized readers to fetch sensitive HDF5 files or enumerate private storage locations.

## Unintended audience

Developers with host visibility, CI administrators, support staff reading logs, anyone with access to crash bundles, and attackers who capture process or environment state.

## Loss / impact

A secondary secret leak becomes a primary privacy breach by granting direct access to otherwise protected datasets.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**
Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**4 - Major.** Exposed credentials can unlock broad access to sensitive object stores, including historical artifacts beyond the immediate workflow.

### Likelihood

**3 - Possible.** Secrets often pass through developer tooling and automation paths that are convenient but poorly scrubbed.

**Risk:** **12 (High)**

## Detection

Inspect CLI usage, environment propagation, config-file handling, CI logs, and crash-report contents for secrets. Review whether standard credential providers or short-lived role-based access are used consistently.

## Controls/mitigations

### Minimization

Prefer instance roles, workload identity, or short-lived tokens over static secrets. Scope each credential to the minimum object set and operation needed.

### Redaction

Remove or mask secrets from logs, trace output, error reports, and telemetry. Prohibit credential passing on the command line where it can surface in process listings and shell history.

### Retention / packaging

Keep secrets out of packaged configs and test fixtures. Set short retention for logs that may contain authentication context and rotate on suspected exposure.

### Review gate

ROS3 and related tooling shall not be approved for production use until secret-injection paths, log-scrubbing rules, and rotation procedures are documented and tested.

## Residual exposure / assumptions

This entry assumes the underlying object store may still log requester identity and object names separately. Secret minimization reduces but does not eliminate privacy exposure from remote access workflows.

## Tests / evidence

- Metadata scan: Check committed configs, scripts, and templates for embedded credentials or signed URLs.
- Artifact inventory: Review process traces, CI logs, crash bundles, and local config paths for secret retention.
- Checklist or release note: Record the approved credential source, redaction behavior, and rotation owner for the workflow.
