# EXP-006: Plugin and filter code can exfiltrate data

**HDF5 SSP vulnerability tags:** `EXT|PRV|OPS`

**Exposure family:** P4 Unintended inclusion during processing

**Lifecycle stage:** Processing and conversion

**Sensitive asset:** Raw data, metadata, and derived values passed through filter and plugin execution paths.

## Trigger

An application enables plugins or filters from untrusted or weakly governed locations, often via environment-variable-controlled discovery such as `HDF5_PLUGIN_PATH`.

## Exposure surface

Executable plugin code, plugin search paths, generated temp files, plugin-specific logs, network calls initiated by the plugin, and transformed outputs written outside the expected workflow.

## Disclosure

A malicious, compromised, or overly chatty plugin can copy, log, or transmit sensitive data and metadata to locations outside the intended processing boundary.

## Unintended audience

Plugin authors, compromised build or runtime environments, operators of exfiltration endpoints, and anyone with access to plugin-generated logs or side outputs.

## Loss / impact

Trusted data-processing paths become covert export channels, undermining assumptions that sensitive data stays local to the HDF5 application.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**
Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** Executable plugin code can access payloads and metadata directly and exfiltrate them with little visibility if not tightly controlled.

### Likelihood

**2 - Unlikely.** Many deployments do not rely on arbitrary plugins, but when they do, environment-based discovery and weak governance create a real path to abuse.

**Risk:** **10 (Moderate)**

## Detection

Inventory enabled plugins, review runtime search paths, and monitor plugin-generated files, logs, and outbound connections. Validate that production services ignore unapproved environment overrides.

## Controls/mitigations

### Minimization

Disable plugins by default and enable only the specific filters required by the workload. Reduce the set of data and metadata passed through extensible paths where possible.

### Redaction

Ensure plugin logs and side outputs do not record payload samples, identifiers, or sensitive metadata. Treat any plugin-produced artifact as potentially sensitive until reviewed.

### Retention / packaging

Lock plugin directories, use signed or verified plugin packages, and prevent ad hoc plugin injection through deployment packaging or runtime environment variables.

### Review gate

No plugin or filter is approved for sensitive workloads until its provenance, allowed behaviors, output artifacts, and search-path controls are documented and tested.

## Residual exposure / assumptions

This entry assumes plugin execution has broad access once loaded. Sandboxing and allowlisting reduce the risk but cannot fully compensate for a plugin trusted with sensitive input.

## Tests / evidence

- Metadata scan: Review plugin configuration, output schemas, and sidecar metadata for unexpected fields or identifiers.
- Artifact inventory: Enumerate plugin binaries, search paths, temp outputs, logs, and any network destinations used during execution.
- Checklist or release note: Record the approved plugin allowlist, packaging source, and environment-variable policy.
