# EXP-008: External links may dangle and leak internal targets

**HDF5 SSP vulnerability tags:** `FMT|PRV|OPS`

**Exposure family:** P2 Structural or on-disk leakage

**Lifecycle stage:** Sharing and publication

**Sensitive asset:** Internal hostnames, filesystem paths, bucket names, endpoints, and object identifiers referenced through external links.

## Trigger

Files are created or exported with external links whose targets are not validated, normalized, or stripped before sharing.

## Exposure surface

External-link metadata embedded in the HDF5 file, along with any resolver behavior, error messages, or diagnostic output that exposes the target path or endpoint when the link is followed.

## Disclosure

Recipients can inspect embedded targets directly or trigger resolution attempts that reveal internal path structure, service names, or protected object locations.

## Unintended audience

External collaborators, downstream archive users, tooling that indexes link targets, and operators reviewing link-resolution failures.

## Loss / impact

Internal topology and identifiers leak outside the intended trust boundary, and link resolution may probe systems that were never meant to be disclosed.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**
Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**4 - Major.** External links can reveal concrete internal locations that help map systems, identify users, or expose confidential project organization.

### Likelihood

**2 - Unlikely.** Not every workflow uses external links, but when they do, validation is often weak and sharing can happen long after creation.

**Risk:** **8 (Moderate)**

## Detection

Scan files for external links, inspect target formats against an allowlist, and test link-resolution behavior in a least-privilege environment to see what paths and errors are exposed.

## Controls/mitigations

### Minimization

Avoid external links in artifacts intended for external distribution. Use internal identifiers or packaging conventions that do not preserve sensitive paths.

### Redaction

Rewrite, normalize, or remove link targets before release. Suppress detailed target strings in error messages and logs shown to broad audiences.

### Retention / packaging

Package shared outputs as self-contained products where possible. Run link resolution and validation in a sandbox so accidental target disclosure does not widen host access.

### Review gate

Any file containing external links shall fail publication review unless each target has been validated, approved, and documented for the intended audience.

## Residual exposure / assumptions

This entry assumes the target may already be inaccessible. Privacy exposure still exists because the target string and resolver behavior can disclose sensitive internal information even when the link dangles.

## Tests / evidence

- Metadata scan: Enumerate all external links and compare target schemes and paths against an allowlist.
- Artifact inventory: Review exported files, error logs, and resolver traces that may preserve link targets.
- Checklist or release note: Record whether links were removed, normalized, or explicitly approved for the release.
