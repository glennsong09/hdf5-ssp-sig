# EXP-004: Remote storage VFDs expand the privacy boundary

**HDF5 SSP vulnerability tags:** `EXT|PRV|OPS`

**Exposure family:** P7 Misplaced trust in privacy by format

**Lifecycle stage:** Transfer and replication

**Sensitive asset:** HDF5 object contents, metadata, and access context handled through remote storage services.

## Trigger

An HDF5 workflow uses remote-storage VFDs such as ROS3 or S3-backed access and assumes the file’s privacy posture is unchanged from local-file use.

## Exposure surface

Bucket and object permissions, network routes, endpoint configuration, access logs, caching layers, mirrored regions, and service-side copies that now sit inside the privacy boundary.

## Disclosure

Sensitive files or access patterns can be exposed through misconfigured object storage, public-read defaults, overly broad credentials, or network paths outside the intended trust zone.

## Unintended audience

Cloud account peers, storage administrators, logging and monitoring teams, compromised service endpoints, and external parties who gain access through misconfigured buckets or objects.

## Loss / impact

The privacy boundary silently widens beyond the application host, increasing the chance of oversharing, unintended replication, and weak auditability.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**
Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**4 - Major.** A remote-storage misconfiguration can expose entire files and their access context across organizational or public boundaries.

### Likelihood

**3 - Possible.** Remote access introduces multiple new control points, and bucket, endpoint, and credential policies are frequently configured outside the application team.

**Risk:** **12 (High)**

## Detection

Review VFD-specific deployment documentation, bucket policies, object ACLs, private-network settings, and access logging. Verify that remote paths used by tests and production stay inside the intended trust boundary.

## Controls/mitigations

### Minimization

Keep only the objects needed for the remote workflow in the bucket, use neutral object keys, and separate public and private datasets into different storage boundaries.

### Redaction

Publish remote-readable derivatives only after metadata and path review. Remove unnecessary identifying information from object names and service metadata.

### Retention / packaging

Use private networking, block public access, restrict replication targets, and document where service-managed copies and logs are retained.

### Review gate

Remote-storage VFD use shall not be approved until the workflow names its storage boundary, credential scope, logging behavior, and object-retention controls.

## Residual exposure / assumptions

This entry assumes the remote service is part of the privacy boundary. Even with correct ACLs, service logs and replicas may still carry sensitive identifiers unless separately governed.

## Tests / evidence

- Metadata scan: Check object keys, service metadata, and exported configuration for sensitive identifiers.
- Artifact inventory: List buckets, prefixes, replicas, cache layers, and log destinations touched by the remote workflow.
- Checklist or release note: Record the approved endpoint, bucket policy baseline, and least-privilege credential scope.
