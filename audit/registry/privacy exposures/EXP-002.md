# EXP-002: Metadata leakage even if payload is encrypted

**HDF5 SSP vulnerability tags:** `FMT|PRV|OPS`

**Exposure family:** P1 Semantic metadata exposure

**Lifecycle stage:** Sharing and publication

**Sensitive asset:** Object names, attributes, labels, hierarchy, and descriptive metadata that reveal meaning without reading the raw payload.

## Trigger

Teams encrypt raw values or rely on coarse storage controls while leaving names, attributes, structure, or schema descriptions readable.

## Exposure surface

Dataset and group names, attribute values, enum labels, committed datatype names, object counts, and hierarchy structure that remain visible in the file or exported metadata views.

## Disclosure

Sensitive meaning such as subject identity, study purpose, device identity, site names, or protected categories can be inferred even when the main data bytes are not intended for disclosure.

## Unintended audience

Anyone allowed to inspect metadata, catalog systems that index file structure, recipients of shared files, and reviewers who only need operational access rather than content visibility.

## Loss / impact

Partial protection fails: observers can infer sensitive context, re-identify records, or learn proprietary study details despite payload encryption.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**
Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**4 - Major.** Metadata alone can reveal regulated or strategically sensitive meaning and can enable re-identification even without direct payload access.

### Likelihood

**3 - Possible.** HDF5 is self-describing, and descriptive names and attributes are common unless teams deliberately minimize them.

**Risk:** **12 (High)**

## Detection

Run metadata scans and schema reviews that inspect names, attributes, labels, and hierarchy for direct identifiers, unnecessary free text, and sensitive semantic clues. Review publication pipelines for metadata exports and indexes.

## Controls/mitigations

### Minimization

Use opaque identifiers, controlled vocabularies, and allowlists for attributes. Keep sensitive semantics out of names, labels, and structural descriptions unless strictly required.

### Redaction

Strip or rewrite sensitive names and attributes before publication. If necessary, generate a public derivative file with neutralized metadata and a separate private mapping.

### Retention / packaging

Avoid publishing metadata-only previews or catalogs derived from sensitive files unless they have passed the same privacy review as the full artifact.

### Review gate

Release workflows shall reject files whose names, attributes, or structure expose direct identifiers, protected categories, or proprietary study details beyond the intended audience.

## Residual exposure / assumptions

This entry assumes payload encryption or access control may exist, but that protection is not sufficient on its own. Structural and correlation risks remain even after metadata redaction.

## Tests / evidence

- Metadata scan: Inspect object names, attributes, labels, and hierarchy against a denylist and an allowlist.
- Artifact inventory: Review exported metadata indexes, previews, and sidecar summaries generated from the file.
- Checklist or release note: Document what metadata was retained, removed, or transformed for the target audience.
