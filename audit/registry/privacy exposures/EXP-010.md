# EXP-010: User block can hide arbitrary data

**HDF5 SSP vulnerability tags:** `FMT|PRV|OPS`

**Exposure family:** P4 Unintended inclusion during processing

**Lifecycle stage:** Sharing and publication; archive, backup, and restore

**Sensitive asset:** Arbitrary bytes stored in the user block, including PII, secrets, manifests, or stale notes not visible through normal object traversal.

## Trigger

Workflows populate the user block intentionally or inherit non-zero user-block bytes from templates, converters, or previous processing steps without reviewing the content before distribution.

## Exposure surface

The user block preceding the HDF5 superblock, copied artifacts, archives, and bytestream-level inspection tools that reveal data outside standard HDF5 object enumeration.

## Disclosure

Sensitive material rides along with otherwise valid HDF5 files and is missed by reviewers who inspect only groups, datasets, and attributes.

## Unintended audience

File recipients, archive operators, forensic analysts, and tooling that extracts raw bytes or packages complete file images for debugging or release.

## Loss / impact

Hidden data bypasses normal privacy checks, causing silent publication of arbitrary sensitive content embedded in the file container.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**
Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**3 - Moderate.** The user block can carry materially sensitive content, but the issue is limited to workflows that use or inherit non-zero user blocks.

### Likelihood

**2 - Unlikely.** Many files have no user block, yet the feature is obscure enough that populated blocks can escape normal review when present.

**Risk:** **6 (Moderate)**

## Detection

Inspect file headers for non-zero user-block sizes, review the raw bytes when a user block exists, and add CI checks that flag unexpected content before publication or archival.

## Controls/mitigations

### Minimization

Disallow user blocks in regulated or externally shared datasets unless there is a documented, approved use case.

### Redaction

Clear or overwrite user-block contents before release, or regenerate the file without a user block when the extra bytes are not required.

### Retention / packaging

Ensure packaging and archive workflows inspect the full file image, not only HDF5 objects. Do not rely on metadata traversal alone to certify privacy review completeness.

### Review gate

Any file with a non-zero user block shall fail release review until the bytes are inspected, justified, and either removed or explicitly approved.

## Residual exposure / assumptions

This entry assumes user-block bytes may not be interpreted by standard HDF5 tooling. Privacy review therefore has to include raw-file inspection or an explicit policy forbidding user blocks.

## Tests / evidence

- Metadata scan: Verify user-block size and flag any file with a non-zero value for deeper inspection.
- Artifact inventory: Track templates, generated outputs, and archived file images that may propagate hidden user-block content.
- Checklist or release note: Record whether user blocks are forbidden, inspected, or intentionally retained for the workflow.
