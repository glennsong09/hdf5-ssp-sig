# EXP-003: Data remanence after dataset deletion

**HDF5 SSP vulnerability tags:** `FMT|LIB|PRV|OPS`

**Exposure family:** P6 Persistence beyond intended lifetime

**Lifecycle stage:** Processing and conversion; archive, backup, and restore

**Sensitive asset:** Raw data and metadata that users believe were deleted from the file.

## Trigger

Applications delete datasets, unlink objects, or rewrite files in place and assume the removed content is securely erased.

## Exposure surface

Free space, file slack, snapshots, backups, replicated copies, and pre-repack artifacts that can still contain recoverable sensitive bytes or stale metadata.

## Disclosure

Sensitive content can be recovered from the file body or downstream copies after deletion, defeating retention limits, right-to-be-forgotten requests, or internal deletion policies.

## Unintended audience

Forensic analysts, storage and backup operators, recipients of archived copies, and any party with low-level file access or snapshot visibility.

## Loss / impact

Deleted data remains recoverable longer than intended, creating regulatory noncompliance and a false sense of secure erasure.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**
Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**4 - Major.** Retained recoverable data can negate deletion commitments and expose information that stakeholders reasonably believed was removed.

### Likelihood

**3 - Possible.** HDF5 deletion does not imply secure wiping, and many teams do not validate post-delete file contents or downstream retention behavior.

**Risk:** **12 (High)**

## Detection

Test deletion workflows by inspecting free space, repacking to fresh files, and checking whether deleted content survives in snapshots, backups, temp directories, or prior generations.

## Controls/mitigations

### Minimization

Prefer short-lived rolling files or partitioned datasets so sensitive content can be retired by destroying whole artifacts rather than deleting in place.

### Redaction

When deletion is required, rebuild or repack into a fresh file that contains only the retained content, then destroy superseded copies using approved media-handling procedures.

### Retention / packaging

Align backup, archive, and snapshot policies with stated deletion semantics. Document whether secure erase is impossible within a given workflow and prevent misleading promises.

### Review gate

Any workflow that claims record deletion or privacy-driven removal shall provide verified evidence for file regeneration, superseded-copy destruction, and backup retention handling.

## Residual exposure / assumptions

This entry assumes lower layers may still keep snapshots or replicas after an HDF5-level repack. Secure deletion remains dependent on the surrounding storage system and lifecycle controls.

## Tests / evidence

- Metadata scan: Verify that deleted object names and attributes do not reappear in regenerated or repacked outputs.
- Artifact inventory: Enumerate active files, snapshots, backups, and temp artifacts that may still contain pre-delete content.
- Checklist or release note: Record the deletion method used, the regenerated artifact identifier, and the disposition of superseded copies.
