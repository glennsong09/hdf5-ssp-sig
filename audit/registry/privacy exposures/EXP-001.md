# EXP-001: No standard whole-file encryption by default

**HDF5 SSP vulnerability tags:** `FMT|PRV|OPS`

**Exposure family:** P7 Misplaced trust in privacy by format

**Lifecycle stage:** Sharing and publication; transfer and replication

**Sensitive asset:** Raw dataset payloads, derived results, and any identifiers stored anywhere in the file.

## Trigger

Sensitive HDF5 files are stored, copied, or transmitted without external encryption because users assume the format itself provides whole-file confidentiality.

## Exposure surface

The entire file at rest or in transit, including payload bytes, metadata, backups, replicated copies, and any intermediate cache or staging location that handles the plaintext file.

## Disclosure

Anyone with storage, host, or network visibility can read the file contents if the surrounding platform does not enforce encryption and authenticated access.

## Unintended audience

Storage operators, cloud administrators, backup operators, incident responders, recipients of misrouted copies, and network observers when transport protection is missing.

## Loss / impact

Direct disclosure of sensitive data at full fidelity, followed by compliance failures, contractual exposure, and loss of control over copied plaintext artifacts.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**
Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**5 - Critical.** If a sensitive file is exposed without encryption, the whole payload and all metadata can be disclosed immediately.

### Likelihood

**4 - Likely.** Whole-file encryption is not a standard HDF5 default, so privacy depends on surrounding storage and transport controls that teams often apply inconsistently.

**Risk:** **20 (Very High)**

## Detection

Review storage-class defaults, object-store policies, transport settings, and release procedures to confirm that encryption at rest and in transit is enforced for every sensitive workflow. Audit exported artifacts and backup paths for plaintext copies.

## Controls/mitigations

### Minimization

Store only the data needed for the intended audience, and keep sensitive datasets in separate files or products so strong encryption requirements can be applied selectively.

### Redaction

Remove unnecessary identifiers and sensitive free text before external sharing so a plaintext exposure does not automatically reveal the full operational context.

### Retention / packaging

Require encrypted storage, encrypted backups, and TLS-protected transfer. Package sensitive HDF5 files only through controlled channels with central key management and auditable access.

### Review gate

No sensitive HDF5 artifact is released, replicated, or archived until the owning workflow documents the encryption boundary, key-management owner, and verification evidence.

## Residual exposure / assumptions

This entry assumes encryption is provided by the surrounding platform rather than the HDF5 format. Even with strong payload protection, metadata and inference risks still need separate review.

## Tests / evidence

- Metadata scan: Confirm the file classification and sensitivity label match the encryption policy applied to the artifact.
- Artifact inventory: Verify that plaintext copies do not remain in staging, backup, test, or release locations outside the approved encryption boundary.
- Checklist or release note: Record where encryption at rest, transport protection, and key ownership are enforced for this workflow.
