# EXP-007: Object-store access logging can leak identifiers

**HDF5 SSP vulnerability tags:** `EXT|PRV|OPS`

**Exposure family:** P6 Persistence beyond intended lifetime

**Lifecycle stage:** Transfer and replication; archive, backup, and restore

**Sensitive asset:** Object names, requester identity, timestamps, and access patterns associated with sensitive HDF5 artifacts.

## Trigger

Remote object access is logged by default or with verbose settings while object names and request context still contain meaningful identifiers.

## Exposure surface

Service access logs, SIEM pipelines, analytics exports, billing views, support dashboards, and retained audit archives that preserve identifiers and behavioral traces.

## Disclosure

Observers learn which objects exist, who accessed them, and when or how often they were read, which can reveal study membership, operational schedules, or internal identifiers.

## Unintended audience

Logging administrators, security analysts without a need to know dataset semantics, vendors handling observability pipelines, and anyone with access to exported log archives.

## Loss / impact

Operational telemetry becomes a separate privacy dataset that can reveal sensitive identifiers and usage patterns long after the file access event.

## Risk assessment

Risk scoring used: **Severity (1-5) x Likelihood (1-5) = Risk (1-25)**
Risk bands: **1-5 Low, 6-10 Moderate, 11-15 High, 16-20 Very High, 21-25 Critical**

### Severity

**3 - Moderate.** Logs usually expose context rather than full payloads, but that context can still be enough to identify subjects, sites, or confidential workflows.

### Likelihood

**3 - Possible.** Object-store logging is common, and object keys are often descriptive unless teams explicitly neutralize them.

**Risk:** **9 (Moderate)**

## Detection

Inspect access-log formats, object-key naming conventions, log retention periods, and downstream analytics exports. Verify what fields are emitted for ROS3 or related remote-access workflows.

## Controls/mitigations

### Minimization

Use neutral or tokenized object names and avoid embedding direct identifiers in bucket paths and keys. Log only the fields required for operations and compliance.

### Redaction

Hash, tokenize, or suppress object identifiers and requester metadata before logs reach broad observability systems where possible.

### Retention / packaging

Treat access logs as sensitive artifacts with restricted access and defined retention limits. Keep performance telemetry separate from production identifiers.

### Review gate

Remote-access workflows shall document what request metadata is logged, who can read it, and how identifiers are reduced before deployment approval.

## Residual exposure / assumptions

Some audit logging may be mandatory. In that case, privacy review shifts to minimizing identifiers, restricting access, and limiting log lifetime rather than removing logging entirely.

## Tests / evidence

- Metadata scan: Review bucket names, object keys, and emitted log fields for direct identifiers and sensitive semantics.
- Artifact inventory: List logging sinks, analytics exports, retained archives, and personnel with access to those records.
- Checklist or release note: Record the logging defaults, redaction settings, and retention period approved for the workflow.
